# Gate2 事前登録 acceptance check

> **status: active（2026-07-22）** — Gate2（NC 490bc99 / 567 pending / モードA群）のフル時系列再シミュを実行する際に、**実行前に読むこと**。

## この文書の目的

Gate2 は全選手・全期間を再シミュするため、日常の `--mode=data-correction`（ripple を delta=0 に強制する運用）では表に出ない変動が一斉に出る。そのとき「原因不明の移動」に見えるが実際には**既知の修正が意図どおり効いているだけ**の項目を、事前に登録しておく。PR-3 の attribution レポート（`src/lib/mnewsRating/rankAttribution.ts`）で mover 会計を突き合わせる際、ここに載っているものは説明済みとして扱ってよい。

ここに無い移動が出た場合は未知の変動なので、止めて調査すること。

**この文書には内部数値（資格判定の閾値実測表・選手別の試合数等）を書かない。** 公開ランキングはレート数値を意図的に非公開にし、ブラックボックスとして提示している。閾値表や選手ごとの空白前後の試合数はその資格ロジックの内部数値そのものであり、Publicリポジトリに置くとその設計を自ら崩す。66.0kgのような**RIZIN公式が既に公開している事実**、および「誰の何が動くか」という**定性的な期待**は載せて構わない。数値が必要な場合は、下記手順で**再実行して自分で再生成する**こと。

## 2. 確立ベテラン例外の閾値分離（PR #188、2026-07-22）

### 何をしたか

`resolveEligibilityScope` の確立ベテラン例外（活動空白カットオフを無効化する判定）が、閾値に `ELIGIBILITY_MIN_FIGHTS`(=3) を流用していた。これは「掲載資格の試合数バー」であって「確立選手の定義」ではなく、値が3で一致していたのは偶然の結合。専用定数 `ELIGIBILITY_ESTABLISHED_VETERAN_MIN_FIGHTS` へ分離した（採用値はコード参照。ここには書かない）。

### 閾値表・選手別の実測値はここに書かない

実測時の閾値表（3/4/5/6での `(iv)` の成否・掲載差分）と、対象選手ごとの空白前後の試合数は、PR #188 のレビュー記録（GitHub PR本文・レビュー履歴）に残っている。これらは**再実行すれば同じ表が再生成できる派生データ**なので、数値そのものではなく再現手順をここに残す。

### 閾値表を再現する手順

1. `src/lib/mnewsRating/constants.ts` の `ELIGIBILITY_ESTABLISHED_VETERAN_MIN_FIGHTS` を対象の値に書き換える。
2. `npx tsx scripts/update-mnews-rating.ts --mode=data-correction` を実行し、`data/rankings.json` を生成する。
3. `git diff -- data/rankings.json` で、現在の採用値（コード参照）で生成した状態との掲載結果差分（REMOVED/ADDED/RANK/RECORD）を見る。
4. `git checkout -- data/` で書き戻し、値を次の候補に変えて2〜3を繰り返す。
5. `npx tsx scripts/test-mnews-rating.ts` を実行し、`(iv)` の成否も併せて記録する。
6. 選手ごとの「空白前後の試合数」と「例外が発動するか」を知りたい場合は、`eligibilityRules.ts` の `resolveEligibilityScope` の判定（`detectInactivityGapCutoff` と `preGapFights >= 閾値` の比較）をread-onlyスクリプトで再現し、`summarizeBoutsForFighter` の結果に対して閾値を差し替えながら評価する（PR #188 作成時に使ったのと同種の使い捨てスクリプトで、リポジトリには残していない）。

### 採用値の選び方（定性の判断基準）

`(iv)` を緑に戻す最小値を起点に、**出力の一致だけでなく因果構造（なぜ掲載されているか）を保存する値**を優先する。閾値を上げすぎると、一部の選手が「例外で救われている選手」から「たまたま別ルートで通っている選手」に静かに変わり、今日の出力は同じでも次に1戦こなしたときの挙動が変わりうる。この基準で候補を絞ったうえで、掲載結果の差分がゼロであることを手順どおり確認して値を決めること。

### Gate2 で予期される移動

- **今日時点の出力は完全に不変**（`--mode=data-correction` の diff は `updatedAt` のみ）。
- ただし**フル時系列再シミュでは資格判定の入力が変わる**。時系列上の中間状態で空白前の試合数がちょうど閾値の選手がいれば、その時点での資格が変わりうる（＝レート推移の分岐点が変わる）。
- 現在の本番データにその境界に該当する選手はいない（実測済み・上記手順で再確認可能）。しかし過去の任意時点ではこの限りではないため、Gate2 で資格ステータスの移動が出た場合はまずここを疑うこと。

### 運用上の注意: `(iv)` を「近代化」しないこと

`scripts/test-mnews-rating.ts` に追加した境界テスト4本（閾値ちょうど／閾値−1／中村大介型／ケース型）は、`ELIGIBILITY_ESTABLISHED_VETERAN_MIN_FIGHTS` **定数からの相対**で期待値を導出しているため、閾値を変更しても自動追随する。

一方、既存の `(iv)`（18ヶ月超の活動空白のテスト）は**空白前3戦という固定フィクスチャ**のままで、これが**絶対値の下限を固定する役割**を担っている。実際に閾値を旧値（3）へ戻すと `(iv)` だけが単独で赤に戻ることを確認済みで、境界テスト4本は定数相対なので緑のまま動かない。

つまり「相対を守るテスト」と「絶対値のピンを打つテスト」の両方があって初めて、閾値の意図しない移動を検知できる関係になっている。**将来 `(iv)` を境界テストと同じ「定数相対」の書き方に書き直す（近代化する）と、絶対値のピンが外れ、閾値が黙って動ける状態に戻る。** テストの意図がコメントで残っていないと次の人が善意でやってしまうため、`(iv)` を触る際は必ずこの関係を踏まえること。

## 付記: `aliases` を追加するときの注意（2026-07-22、PR #186 で判明）

`fighters.ts` の `aliases` は**ランキング集計層だけでなく、対戦相手名のリンク解決（`findFighterSlugByName`）も消費している。**

PR #186 で「黒薔薇くん」を別名登録した結果、選手ページの戦績テーブル上で対戦相手名のテキストは「黒薔薇くん」のまま変わらなかったが、**そこが `/fighters/suzuki-hiroaki` へのリンクになった。** 今回は同一人物へのリンクなので正しい挙動だったが、**`rankings.json` を比較するだけの dry-run ではこの変化を捕捉できなかった。**

したがってエイリアスを追加する際は、以下を手順に含めること。

1. **事前に `aliases` の全消費側を洗う**（`grep -rn "aliases" src/` で、名前解決層・リンク解決層・その他の参照を列挙する）。
2. 各消費側について、その別名が入ることで**読者に見える表示・リンクがどう変わるか**を確認する。
3. `rankings.json` の diff だけを根拠に「影響なし」と結論しない。

同種の教訓として、`ELIGIBILITY_MIN_FIGHTS` の流用（PR #188）も「1つの定数を複数の概念が共有していた」問題である。**共有されている値・機構に手を入れるときは、まず消費側を数えること。**
