# 指示書: 夢のカード（/dream）と X投稿用カード（/vs）の統合

版: v1.0（2026-07-16）
関連文書: `vs-card-spec.md`（VS対戦比較カード リデザイン仕様書 v1.0）— カードの見た目・バー計算・名前の高さ対策はすべて同仕様に従う

---

## 0. 現状と問題

| ページ | 現状 | 問題 |
|---|---|---|
| `/dream` | 選手A/Bセレクタ + リッチ比較（戦績・直近5戦・共通対戦相手）+ Xシェア | シェア先が自ページでなく `/vs` に飛ぶ。URLに選択状態が乗らないため、画面の組み合わせと共有URLがズレ得る |
| `/vs/{a}/{b}` | OGP画像を `<img>` 表示 + X投稿ボタン + 選手リンク2件のみ | 中身が画像だけの行き止まり。Xから流入したユーザーに比較データも回遊導線もない。同一機能の重複実装 |

**統合方針: `/vs/{slugA}/{slugB}` を唯一の対戦比較ページに昇格し、`/dream` をその入口（セレクタUI）にする。カード描画コンポーネントとOGP生成ロジックは1系統に統一する。**

## 1. ルーティング設計

### 1.1 `/vs/{slugA}/{slugB}` — 正式な対戦比較ページ
- 新VSカード（vs-card-spec.md準拠）をフル描画: 階級ラベル / 選手名帯 / 綱引きバー×2 / 直近5戦 / 過去対戦 / 共通対戦相手
- 現行の「OGP画像を<img>で貼る」実装は**廃止**。OGPはmetaタグ専用に戻す
- ページ下部に回遊導線: 両選手の個別ページ / 関連ニュース（両選手のタグが付いた記事があれば） / 「別のカードを作る → /dream」
- 「選手を入れ替えて再選択」ボタン → `/dream?a={slugA}&b={slugB}` で初期選択済みセレクタに戻す

### 1.2 スラッグ順の正規化【重要】
- `/vs/a/b` と `/vs/b/a` の重複URLを許さない。**正規順 = スラッグの辞書順昇順**とする
- 非正規順アクセスは正規URLへ **308恒久リダイレクト**（サイト方針として301指示は308でOK）
- canonical タグも正規順URLを指す
- カード内の左右表示順は正規順に従う（左=辞書順で先のスラッグ=赤コーナー）

### 1.3 `/dream` — セレクタページ
- 役割: 選手A/B選択UI + プレビュー。比較カード自体は `/vs` と同一コンポーネントをインライン描画
- 選択確定時に `history.replaceState` で `?a=&b=` をURLに反映（リロード・共有時に選択状態を復元）
- 「このカードのページへ」リンクで `/vs/{a}/{b}` へ遷移可能に
- シェアボタンは現行どおり `/vs` の正規URLを共有（画面と共有先の内容が同一コンポーネントなのでズレなくなる）

### 1.4 リダイレクト・後方互換
- 既存の `/vs/{a}/{b}` URL（X上に既に流通）はそのまま新ページとして機能する。パス変更なし
- 旧「X投稿用カード」への内部リンク・ナビゲーションがあれば `/dream` または `/vs` へ差し替え、旧ルートが別パスに存在する場合は308

## 2. コンポーネント統合

```
components/
  vs/
    VsCard.tsx          … カード本体（vs-card-spec.md §3-6を実装）
    VsCard.themes.ts    … light / dark トークン（spec §2）
    vsMath.ts           … shareA計算・クランプ・fighterNameSize()（spec §4.2, §5.1）
app/
  vs/[a]/[b]/page.tsx   … VsCard(light) + 回遊導線
  dream/page.tsx        … セレクタ + VsCard(light) インライン
  api/og/vs/[a]/[b]/route.tsx … Satori。VsCard darkと同一のvsMath関数を使用
```

- **vsMath.ts をWeb/OGPの単一ソースにする。** 数値計算・フォント段階のロジックをOGP側に複製しない
- 既存の `/api/og/vs/...?v=4` は新デザイン（spec §7 ダーク版）で再生成し、`?v=5` にバンプしてXのカードキャッシュを更新

## 3. シェア文言の統一

- 現状 `/dream` は「もし「A vs B」が実現したら―― #夢のカード」付き、`/vs` ページ内ボタンはURLのみで文言なし。**前者に統一**
- intent仕様:
  `text = もし「{選手A} vs {選手B}」が実現したら―― #夢のカード`
  `url = https://www.mnews.jp/vs/{slugA}/{slugB}`（正規順）
- 実カード（過去対戦あり・同階級の現実的マッチ）でも文言は共通でよいが、将来イベント連動時に差し替えられるよう文言は1箇所の定数に置く

## 4. SEO・インデックス方針

組み合わせは選手数nに対しn(n-1)/2件（現状140人超 → 約1万URL）発生する。全量インデックスは薄いプログラマティックページの大量生成としてコアアップデートで叩かれるリスクが高い（既に「MMAニュース」で変動を経験済み）。

- **デフォルト: `noindex, follow`**
- 以下のいずれかを満たすページのみ index 許可:
  1. 過去対戦が1回以上ある
  2. 共通対戦相手が1人以上いる
  3. 同一団体かつ同一階級（現実的マッチ）
- sitemap には index対象のみ載せる
- タイトル: `{選手A} vs {選手B} 対戦比較｜戦績・共通対戦相手 - Mニュース`
- meta description: 現行の「A（x勝y敗）vs B（w勝z敗）の対戦カード。」に共通対戦相手数を追記

## 5. データ前提（vs-card-spec.md §8 と同一）

- 重複スラッグの名寄せ解決を先行実施（karshyga-dautbek / dautbek-karshyga 等）。未解決のまま /vs を量産すると重複URL・分裂した共通対戦相手がインデックスされる
- 同一相手との複数対戦は共通対戦相手1行に集約し結果ドットを並べる

## 6. 実装順序

1. vsMath.ts 切り出し + VsCard 実装（spec準拠、Storybook等でエッジケース確認）
2. `/vs/[a]/[b]` を新ページに置換 + スラッグ順正規化・301
3. `/api/og/vs` を新デザイン化、`?v=5` バンプ
4. `/dream` をセレクタ+同一コンポーネント構成に改修、URLクエリ同期
5. noindex制御 + sitemap 更新
6. 旧「X投稿用カード」導線の差し替え・削除

## 7. 受け入れテスト

1. `/dream` で選手選択 → シェア → X経由で `/vs` 着地 → **画面内容が/dreamのプレビューと完全一致**
2. `/vs/nakamura-rinya/taira-tatsuro`（非正規順）→ 301 → `/vs/taira-tatsuro/nakamura-rinya`
3. `/vs` 直リンク着地時、比較データ全項目 + 回遊導線（選手ページ・/dreamへ）が表示される
4. X Card Validator で新OGP（v5）が表示される
5. 過去対戦0・共通対戦相手0・別団体別階級の組み合わせ → `noindex` が付与される
6. 過去対戦ありの組み合わせ → index可、sitemapに含まれる
7. `/dream?a=…&b=…` 直アクセスで選択状態が復元される
8. vs-card-spec.md §9 の7ケース（80vs80、長名等）が `/vs` `/dream` 両方で通る
